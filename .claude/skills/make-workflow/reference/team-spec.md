# 팀 스펙(team-spec) 정본 형식

workflow-architect가 산출하고, make-workflow 스킬·skill-author·workflow-integration-reviewer가 소비하는 설계 문서의 형식. 경로: `_workspace/<워크플로명>/team-spec.md`.

## 형식

```markdown
# <워크플로명> 팀 스펙

## 목표
<이 워크플로가 만들어내는 최종 산출물과 성공 기준. 1~3문장.>

## 스킬
- **커맨드명**: <디렉토리명이 됨. 소문자-하이픈>
- **description 초안**: <로드 트리거 중심 한 문단>
- **오케스트레이션 개요**: <단계 순서, 루프 조건, 최대 재시도, 실패 종료. 번호 목록>
- **사용자 개입 지점**: <있으면 명시. 없으면 "없음">

## 에이전트
### <에이전트명 1>   ← 소문자-하이픈, 파일명과 일치
- **역할**: <하는 일 1~2문장>
- **입력**: <프롬프트로 받는 것: 경로, 지시, 계약>
- **출력**: <내놓는 것: 파일 경로와 형식, 또는 보고 형식>
- **상류/하류**: <누가 부르나 → 누구에게 넘어가나>
- **tools 제안**: <최소 권한 목록>
- **model 제안**: haiku | sonnet | opus  (+한 줄 근거)

### <에이전트명 2>
...

## 파이프라인
<에이전트 실행 순서와 루프를 화살표로. 예: generator → tester → reviewer ─(nonpass)→ generator>

## 정합성 포인트
<integration-reviewer가 검증해야 할 지점 목록: 이름 일치, 경로 일치, 형식 일치 등>
```

## 작성 원칙

- **에이전트 수는 최소로.** 역할이 도구·모델·판정 기준 중 하나라도 다를 때만 분리한다. 합쳐도 되는 두 역할은 합친다.
- **I/O 계약이 제일 중요하다.** 상류의 "출력"과 하류의 "입력"이 문자 그대로 맞아떨어져야 한다(같은 경로, 같은 형식).
- **model 제안 기준**: 무거운 생성·설계=opus, 일반 작성·검증=sonnet, 단순·대량·기계적=haiku.
- **재사용 우선.** 이미 있는 에이전트(예: make-agent, make-agent-reviewer)로 되는 역할은 새로 만들지 말고 재사용으로 표기한다.
- 중간 산출물은 전부 `_workspace/<워크플로명>/` 아래 경로로 정한다.
